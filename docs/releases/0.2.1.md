This release is a follow-up to [v0.2.0](https://github.com/Thetanuts-Finance/thetanuts-sdk/blob/main/CHANGELOG.md) — the Base_r12 deployment cutover. ## Why this release exists

After v0.2.0 shipped to `upstream/v0.2.0`, an adversarial code-review pass (OpenAI Codex CLI in challenge mode) was run against the diff. It found 11 production-relevant issues, the most severe of which would silently revert in production once the protocol owner enabled non-zero contract fees. The first pass of fixes was committed; a second adversarial pass against those fixes found 7 more issues, including one P1 in the same class as the original bug. A third pass against the second-round fixes found 3 more (1 P2 + 2 P3) and validated everything else.

**The total tally: 22 issues fixed across 3 review cycles, plus 2 unprompted improvements** (`RANGER_OPTION_ABI` was promised in v0.2.0 docs but never re-exported; documentation drift in 4 README/CHANGELOG locations).

This release exists because v0.2.0 was unsafe to depend on. v0.2.1 is the first stable v0.2.x.

## Quick upgrade guide

```bash
npm install @thetanuts-finance/thetanuts-client@^0.2.1
```

If you adopted v0.2.0, the only required changes are:

```typescript
// 1. Event method renamed
- const events = await client.events.getCollateralReturnedEvents(option);
+ const events = await client.events.getExcessCollateralReturnedEvents(option);

// 2. Event field shape changed (was 3 fields, now 3 different fields)
- events[i].optionAddress      // gone — use BaseEvent.address from filter context
- events[i].seller             // still here
- events[i].amountReturned     // renamed →
+ events[i].collateralToken    // new
+ events[i].collateralReturned // renamed value field

// 3. RangerModule.reclaimCollateral parameter renamed
- await client.ranger.reclaimCollateral(rangerAddress, recipient);
+ await client.ranger.reclaimCollateral(rangerAddress, ownedOption);
//   semantic also changed: the second arg is the option being reclaimed FROM,
//   not a transfer destination. Reclaimed funds go to the caller.

// 4. OptionSplit event has 2 new fields if you were destructuring
- const { newOption, collateralAmount } = splitEvent;
+ const { newOption, collateralAmount, feePaid, counterparty } = splitEvent;

// 5. Butterfly reverse-lookup names
- info.name === 'CALL_FLYS'   // never matches now
+ info.name === 'CALL_FLY'    // matches public ImplementationAddresses key
```

That's it. No other source changes are required to move from v0.2.0 → v0.2.1. The remaining fixes are runtime behavior corrections (ABI shapes, payable plumbing, zero-address guards) — your code calls the same methods and gets correct results.

If you're upgrading from v0.1.x, see the [Migration Guide](../resources/migration-guide.md) — there are more changes to be aware of.

## Production-revert fixes

These are bugs that would silently revert calls in production.

### `split` and `reclaimCollateral` are payable in r12

**Severity:** P1
**Commits:** `441a189`, `e6e6803`, `11aa3b0`

The r12 contracts mark `split(uint256)` and `reclaimCollateral(address)` as `payable`. They consume the on-chain `getSplitFee()` and `getReclaimFee(ownedOption)` as `msg.value`. The v0.2.0 ABIs declared them `nonpayable` and the SDK modules called them with no `value` field.

Symptom in production: every `split` and `reclaimCollateral` call would revert as soon as the contract owner set a non-zero fee. The error would surface deep inside ethers as a generic transaction revert — not a clear SDK-level error.

**Fix:**

- ABIs: `stateMutability: 'nonpayable'` → `'payable'` in `option.ts`, `ranger.ts`, and the `LOAN_OPTION_ABI` string in `loan.ts`.
- `OptionModule.split` and `RangerModule.split` now read `getSplitFee()` from the contract and forward as `msg.value`:

  ```typescript
  const readContract = this.getReadContract(optionAddress);
  const splitFee = await readContract.getSplitFee();
  const tx = await contract.split(splitCollateralAmount, { value: splitFee });
  ```

- `RangerModule.reclaimCollateral` reads `getReclaimFee(ownedOption)` and forwards as `msg.value` (see next item for the ownedOption semantic).

If `getSplitFee()` returns 0 (which it does today), `msg.value: 0n` is forwarded and the call succeeds. The fix is fee-amount-agnostic.

### `getReclaimFee` is keyed on the option, not the caller

**Severity:** P1
**Commits:** `11aa3b0`, `eb034ad`

The r12 `getReclaimFee(address ownedOption)` returns the reclaim fee for a specific option position. The v0.2.0 fix erroneously passed the **signer's address** to `getReclaimFee`. The signer's address is not a valid option contract — the call would return 0, revert, or give a nonsensical fee, and the `msg.value` forwarded to `reclaimCollateral` would not match what the contract expected.

This is the same class of mistake as the original codex-found bug, just at a deeper layer. Caught in the second adversarial pass.

**Fix:**

```typescript
// ❌ before (v0.2.0 + first-pass fix):
const callerAddress = await signer.getAddress();
const reclaimFee = await readContract.getReclaimFee(callerAddress);

// ✅ after (v0.2.1):
this.client.requireSigner();  // still need a signer for the tx itself
const reclaimFee = await readContract.getReclaimFee(ownedOption);
```

ABI input parameter renamed from `caller` to `ownedOption` in both `src/abis/option.ts` and `src/abis/ranger.ts` to prevent the same misreading recurring.

## ABI shape corrections

These don't break runtime calls in v0.2.0 because ethers encodes by position, not by name. But they break **decoding** for static calls and confuse external ABI consumers.

### `OptionBook.getValidNumContracts` returns a tuple

**Severity:** P2
**Commits:** `9f0e6ab`, `5fa2f78`

```diff
- outputs: [{ name: '', type: 'uint256', internalType: 'uint256' }]
+ outputs: [{
+   name: 'result',
+   type: 'tuple',
+   internalType: 'struct OptionBook.ValidContractsResult',
+   components: [
+     { name: 'validContracts', type: 'uint256', internalType: 'uint256' },
+     { name: 'collateralRequired', type: 'uint256', internalType: 'uint256' },
+   ],
+ }]
```

Also: third input parameter renamed `numContracts` → `desiredContracts` to match the canonical r12 name.

Ethers v6 unwraps single-tuple-output functions, so `await book.getValidNumContracts(...)` returns the inner tuple directly. Both `result.validContracts` and `result[0]` work.

### `optionType` is `view` for BaseOption, `pure` for RangerOption

**Severity:** P3
**Commits:** `21526dd`, `96bb25d`

The first-pass fix marked both `BaseOption.optionType` and `RangerOption.optionType` as `pure`. The canonical r12 JSONs say:

- `BaseOption.optionType()` → **`view returns (uint256)`**
- `RangerOption.optionType()` → **`pure returns (uint256)`**

Ethers calls work either way (it doesn't enforce stateMutability on view/pure), but the ABI is now correct against the on-chain artifact.

### `returnExcessCollateral` declares its `uint256` return

**Severity:** P3
**Commit:** `29eb5b7`

```diff
  name: 'returnExcessCollateral',
  inputs: [],
- outputs: [],
+ outputs: [{ name: '', type: 'uint256', internalType: 'uint256' }],
  stateMutability: 'nonpayable',
```

Static-call consumers that expected the returned amount were getting `undefined`.

### `LOAN_COORDINATOR_ABI.assetConfigs` returns a tuple

**Severity:** P2
**Commit:** `0db7187`

```diff
- 'function assetConfigs(bytes32) view returns (bool)',
+ 'function assetConfigs(bytes32) view returns (address collateralToken, address priceFeed, address settlementToken, bool isActive)',
```

The v0.2.0 ABI string declared a single `bool` return; the r12 contract returns four fields. Anyone reading the asset-configs mapping was misdecoding.

## Event shape corrections

The v0.2.0 event ABIs were hand-built and several diverged from the canonical r12 JSONs. **Any consumer of `client.events.*` for these events would silently misdecode logs against r12 contracts.**

### `OptionInitialized` (11 fields, was 3)

**Severity:** P1 for log-decoding
**Commits:** `7f2e5bb`

The event was declared in `RANGER_OPTION_ABI` with 3 fields (`optionAddress`, `creator`, `paramsHash`) and was missing entirely from `BASE_OPTION_ABI`. The actual r12 shape has 11 fields:

```typescript
event OptionInitialized(
  address indexed buyer,
  address indexed seller,
  address indexed createdBy,
  uint256 optionType,
  address collateralToken,
  address priceFeed,
  uint256[] strikes,
  uint256 expiryTimestamp,
  uint256 numContracts,
  uint256 collateralAmount,
  bytes extraOptionData
);
```

Now declared correctly in both `BASE_OPTION_ABI` and `RANGER_OPTION_ABI`.

### `OptionSplit` gained `feePaid` and `counterparty`

**Severity:** P1 for log-decoding
**Commit:** `7f2e5bb`

```diff
  event OptionSplit(
    address indexed newOption,
    uint256 collateralAmount,
+   uint256 feePaid,
+   address indexed counterparty
  );
```

`OptionSplitEvent` type and the `getOptionSplitEvents` extractor were updated to match.

### `TransferApproval` field order corrected

**Severity:** P1 for log-decoding (Ranger only)
**Commit:** `7f2e5bb`

```diff
  event TransferApproval(
-   address indexed optionAddress,
-   bool isBuyer,
-   address indexed target,
-   bool isApproved
+   address indexed target,
+   address indexed from,
+   bool isBuyer,
+   bool isApproved
  );
```

`option.ts`'s ABI was already correct; only `ranger.ts` had the wrong order.

### `OptionSettlementFailed` has no inputs

**Severity:** P1 for log-decoding (Ranger only)
**Commit:** `7f2e5bb`

```diff
- event OptionSettlementFailed(address indexed optionAddress, string reason);
+ event OptionSettlementFailed();
```

The SDK still populates `optionAddress` from filter context for caller convenience — see `src/types/events.ts`.

### `CollateralReturned` renamed to `ExcessCollateralReturned`

**Severity:** P1 for log-decoding + breaking API change
**Commits:** `7f2e5bb`, `9cbad55`

The r12 contracts renamed the event and changed its shape:

```diff
- event CollateralReturned(address indexed optionAddress, address indexed seller, uint256 amountReturned);
+ event ExcessCollateralReturned(address indexed seller, address indexed collateralToken, uint256 collateralReturned);
```

Cascading SDK changes:

- `client.events.getCollateralReturnedEvents` → **renamed to `getExcessCollateralReturnedEvents`**
- `CollateralReturnedEvent` type → **renamed to `ExcessCollateralReturnedEvent`** with new fields
- Field extractor in `events.ts` updated
- Re-export in `src/index.ts` updated

## Safety upgrades

### Zero-address guards on every RFQ entry point

**Severity:** P2
**Commits:** `8d7d8aa`, `5195879`, `eb034ad`

The seven physical multi-leg implementations (`PHYSICAL_CALL_SPREAD`, `PHYSICAL_PUT_SPREAD`, `PHYSICAL_*_FLY`, `PHYSICAL_*_CONDOR`, `PHYSICAL_IRON_CONDOR`) are stored as `0x0000…0000` in the chain registry until those contracts deploy.

v0.2.0 had a single zero-address guard inside `getPhysicalImplementationForStructure` (the helper called by the high-level `buildPhysical*RFQ` builders). But four other public entry points took raw `params.implementation` and bypassed the guard entirely:

- `requestForQuotation` — fixed in `8d7d8aa`
- `encodeRequestForQuotation` — fixed in `8d7d8aa`
- `registerReferral` — fixed in `5195879`
- `callStaticCreateRFQ` — fixed in `eb034ad`

A user reading `chainConfig.implementations.PHYSICAL_CALL_SPREAD` and passing it directly to any of these would have produced calldata or sent a transaction targeting the zero address. Now all four call a private helper:

```typescript
private assertImplementationDeployed(impl: string, where: string): void {
  if (!impl || impl === '0x0000000000000000000000000000000000000000') {
    throw createError(
      'INVALID_PARAMS',
      `${where}: implementation 0x000…000 — physical multi-leg contracts not yet deployed in r12. ` +
      `If this is unexpected, check chainConfig.implementations for your option type.`,
    );
  }
}
```

### `RangerModule` chain guard

**Severity:** P3
**Commits:** `5f971ce`, `01d0184`

`RangerModule` previously constructed cleanly on Ethereum mainnet (which has `implementations: {}`) and any other chain without RangerOption deployed, then failed deep inside ethers with a cryptic `eth_call` error when the user actually called a method.

**Fix:** Mirror the `StrategyVaultModule._disabled` pattern. Set `_disabled` at construction time when `chainConfig.implementations.RANGER` is missing **or** equal to `0x0000…0000` (case-insensitive). Every public method calls `ensureEnabled()` first:

```typescript
private ensureEnabled(): void {
  if (this._disabled) {
    throw createError(
      'NETWORK_UNSUPPORTED',
      `RangerModule requires a chain with RangerOption deployed; chainId ${this.client.chainId} has no RANGER implementation.`,
    );
  }
}
```

Verified to throw cleanly on `chainId 1` (Ethereum) before reaching any contract call.

### Lender filter doesn't break on missing `convertToLimitOrder`

**Severity:** P2
**Commit:** `8379c07`

The r12 contract dropped the `convertToLimitOrder` field from `LoanRequested` events. The r12 indexer is expected to drop it from API responses too.

The previous filter at `src/modules/loan.ts:567` was:

```typescript
if (!loan.convertToLimitOrder) continue;  // skips records where the field is missing
```

Against an r12 indexer that omits the field, this returned an empty list for every caller. Fixed to:

```typescript
// r12 indexer no longer surfaces convertToLimitOrder. Treat missing
// field as eligible; only skip when it's explicitly false.
if (loan.convertToLimitOrder === false) continue;
```

Type also updated: `LoanIndexerLoan.convertToLimitOrder` is now optional.

## Breaking changes from 0.2.0

These are real source-breaking changes for code that already integrated v0.2.0. Mitigated by the fact that v0.2.0 was on npm only briefly.

| Change | Migration |
|---|---|
| `client.events.getCollateralReturnedEvents` → `getExcessCollateralReturnedEvents` | Rename the call site |
| `CollateralReturnedEvent` type → `ExcessCollateralReturnedEvent` | Rename the type import; field shape changed (see below) |
| `ExcessCollateralReturnedEvent` fields | Was `{ optionAddress, seller, amountReturned }`; now `{ seller, collateralToken, collateralReturned }`. `optionAddress` is gone — use `BaseEvent.address` from the filter context. |
| `OptionSplitEvent` shape | Added `feePaid: bigint` and `counterparty: string` fields. Existing destructuring still works for the old fields. |
| `OptionSettlementFailedEvent` shape | The contract event has no inputs. SDK still populates `optionAddress` from filter context. No code change required. |
| `getOptionImplementationInfo(addr).name` for butterflies | Returns `'CALL_FLY'` / `'PUT_FLY'` (was `'CALL_FLYS'` / `'PUT_FLYS'`). Update any string equality checks. |
| `ProductName` union (in `client.utils`) | No longer includes `'CALL_FLYS'` / `'PUT_FLYS'`. Use `'CALL_FLY'` / `'PUT_FLY'`. |
| `RangerModule.reclaimCollateral(rangerAddress, recipient)` | Second parameter renamed to `ownedOption`. Positional callers unaffected; named callers using TypeScript object-shape break. **Semantic also changed** — see below. |
| `RangerModule.reclaimCollateral` semantics | The second arg is now the option being reclaimed FROM, not a transfer destination. Reclaimed collateral goes to the caller (the signer). |
| `RangerContract.optionType()` typed return | `Promise<bigint>` (was `Promise<string>`). Only affects callers reading the typed contract interface directly. |

## Publishing checklist

For maintainers preparing to publish to npm:

1. PR `#14` (`v0.2.1` → `main`) merged to `main`.
2. `git checkout main && git pull --ff-only` — confirm `main` advances to `7233d91` or its merge commit.
3. `git tag -a v0.2.1 -m "v0.2.1 — codex-found r12 fixes" && git push upstream v0.2.1` — tag the release.
4. `npm whoami` — confirm logged in as a user with publish rights to `@thetanuts-finance`.
5. `npm pack --dry-run` — eyeball the tarball contents; should be `dist/` only.
6. `npm publish --dry-run --access public` — final preflight.
7. `npm publish --access public` — go live. 2FA OTP if enabled.
8. `npm view @thetanuts-finance/thetanuts-client@0.2.1` — verify on the registry.
9. After ~1–2 minutes, GitBook syncs and the new `Ranger Options` guide appears at `docs.thetanuts.finance/sdk/guides/ranger`.
